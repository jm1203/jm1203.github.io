## Level 3

### 요약
**컨테이너 웹서버의 proxy 기능을 SSRF로 이용해 ECS 컨테이너 메타데이터/자격증명에 접근하는 문제**

- 문제에서 주어진 정보
    - Target: `http://container.target.flaws2.cloud/`
    - Proxy 기능: `http://container.target.flaws2.cloud/proxy/http://flaws.cloud`

- 힌트1에서 주어진 정보
    >ECS 컨테이너는 자격증명을 169.254.170.2/v2/credentials/**GUID** 에서 얻는다. 여기서 GUID는 컨테이너 환경변수 **AWS_CONTAINER_CREDENTIALS_RELATIVE_URI** 이다.


### proxy -> SSRF
>flaws1.cloud에서 비슷한 문제가 있었다.

- **Proxy**는 별도 AWS 서비스가 아니라 ECS에서 실행 중인 컨테이너 내부 웹앱의 기능이다. 원래 사용자는 ECS 내부 주소에 직접 접근할 수 없다
- 하지만 사용자가 입력한 URL을 제한하지 않으면 **SSRF**가 발생한다.

#### SSRF 확인
- **169.254.170.2**는 ECS 컨테이너 내부에서 접근 가능한 메타데이터/자격증명 엔드포인트다.
- 이 주소를 proxy를 통해 요청해서 정보가 나온다면 SSRF가 가능한 것이다.

![Desktop View](..\assets\img\flaws2.cloud\level3-1.png)

- 메타데이터에 사용할만한 정보는 없어보인다.

### 환경 변수 AWS_CONTAINER_CREDENTIALS_RELATIVE_URI
- 힌트에 따르면 ECS 자격증명 URI의 GUID는 환경변수에 있다.
- Linux에서는 실행 중인 프로세스의 환경변수를 `/proc/self/environ`에서 볼 수 있다. (힌트2)
- 그런데 이 proxy는 **http://**뿐 아니라 **file://**도 처리할 수 있었다. (힌트3)

![Desktop View](..\assets\img\flaws2.cloud\level3-2.png)

- **AWS_CONTAINER_CREDENTIALS_RELATIVE_URI** = /v2/credentials/c286546e-e821-4868-b97c-61274248dba7


### 자격증명 획득
- 환경변수에서 얻은 값을 `169.254.170.2` 뒤에 붙인다.
![Desktop View](..\assets\img\flaws2.cloud\level3-3.png)

- 응답에는 ECS task role의 임시 AWS 자격증명이 나온다.

### 정리
