## 개요

Attacker 코스가 “어떻게 AWS 환경을 침투하는가”를 다뤘다면, Defender 코스는 “침해 이후 어떤 로그가 남고, 그 로그를 어떻게 분석하는가”에 초점이 맞춰져 있다.

Objective 1~3은 본격적인 분석 단계라기보다는, **CloudTrail 로그를 확보하고 분석 가능한 형태로 가공하는 준비 단계**에 가깝다.


```text
Security Account 접속
→ CloudTrail 로그 다운로드
→ Target Account 접근
→ jq를 이용한 로그 필드 추출
→ 공격 타임라인 분석 준비
```


## Objective 1: Download CloudTrail logs

### 요약

**Security Account에 접속하여 Target Account의 CloudTrail 로그를 다운로드하는 단계**

이 단계에서는 AWS CLI를 설정하고, CloudTrail 로그가 저장된 S3 버킷에서 로그 파일을 내려받는다.

---

### 배경

AWS를 사용할 때, 여러 AWS 계정을 운영하는 경우가 많다.

```text
Security Account
Prod Account
Dev Account
Staging Account
```

이때 각 계정의 CloudTrail 로그를 개별 계정에만 저장하지 않고, 별도의 **Security Account** 또는 **Log Archive Account**로 중앙 수집하는 구성을 자주 사용한다.

이 구조를 사용하는 이유는 다음과 같다.

- 침해된 계정에서 공격자가 로그를 삭제하는 것을 방지
- 여러 계정의 로그를 중앙에서 분석
- 사고 대응과 감사 대응 용이
- 로그 위변조 위험 감소

---

### 계정 구조

Defender 코스에서는 두 개의 계정이 등장한다.

| 계정 ID | 역할 |
|---|---|
| `322079859186` | Security Account |
| `653711331788` | Target Account |

Security Account는 보안 분석을 수행하는 계정이고, Target Account는 실제 공격이 발생한 계정이다.

---

### AWS CLI 설정 확인

AWS CLI 설정 후 다음 명령어로 현재 identity를 확인한다.

```bash
aws sts get-caller-identity --profile security
{
    "UserId": "AIDAJXZBU42TNFRNGBBFI",
    "Account": "322079859186",
    "Arn": "arn:aws:iam::322079859186:user/security"
}
```


여기서 중요한 점은 현재 사용 중인 계정이 `322079859186`, 즉 **Security Account**라는 것이다.

---

### S3 버킷 확인

접근 가능한 S3 버킷을 확인한다.

```bash
>aws s3 ls --profile security

2018-11-20 05:54:31 flaws2-logs
```

flaws2-logs 버킷에는 Target Account에서 발생한 CloudTrail 로그가 저장되어 있다.

---

### CloudTrail 로그 다운로드

다음 명령어로 로그를 다운로드한다. 다운로드 후 로그는 다음과 같은 경로에 저장된다.

```bash
> aws s3 sync s3://flaws2-logs . --profile security

AWSLogs/653711331788/CloudTrail/us-east-1/2018/11/28/
```


---

### CloudTrail 경로 구조

CloudTrail 로그 경로는 일반적으로 다음 형식을 가진다.

```text
AWSLogs/<Account ID>/CloudTrail/<Region>/<Year>/<Month>/<Day>/
```

이번 실습의 경로는 다음과 같다.

```text
AWSLogs/653711331788/CloudTrail/us-east-1/2018/11/28/
```

각 요소의 의미는 다음과 같다.

| 경로 요소 | 의미 |
|---|---|
| `AWSLogs` | AWS 로그 기본 prefix |
| `653711331788` | 로그가 발생한 Target Account |
| `CloudTrail` | CloudTrail 로그 |
| `us-east-1` | 이벤트 발생 리전 |
| `2018/11/28` | 로그 날짜 |

여기서 주의할 점은 로그를 다운로드한 계정은 Security Account지만, 로그 경로에는 Target Account ID가 들어간다는 것이다.

즉, **Target Account에서 발생한 CloudTrail 로그가 Security Account의 S3 버킷에 중앙 저장된 구조**다.




## Objective 2: Access the Target Account

### 요약

**Security Account에서 Target Account의 보안 조사용 IAM Role을 AssumeRole하여 Target Account에 접근하는 단계**

Objective 1에서는 로그를 다운로드했다.  
Objective 2에서는 실제 Target Account 내부 리소스를 조사하기 위해 Target Account의 IAM Role을 위임받는다.

---

### 핵심 개념: Cross-Account AssumeRole

AWS에서는 한 계정이 다른 계정의 IAM Role을 임시로 사용할 수 있다.

이를 **Cross-Account AssumeRole**이라고 한다.

이번 실습 구조는 다음과 같다.

```text
Security Account
322079859186
        │
        │ sts:AssumeRole
        ▼
Target Account
653711331788
role/security
```

Security Account가 Target Account의 IAM 정보를 가져오는 것이 아니다.

정확히는 **Target Account 안에 미리 만들어진 `security` Role을 Security Account가 임시로 빌려 쓰는 구조**다.

---

### AWS CLI profile 설정

기존 Security Account profile은 다음과 같다.

```ini
"~\.aws\config"

[profile security]
region=us-east-1
output=json
```

여기에 Target Account 접근용 profile을 추가한다.

```ini
[profile target_security]
region=us-east-1
output=json
source_profile = security
role_arn = arn:aws:iam::653711331788:role/security
```

각 설정의 의미는 다음과 같다.

| 설정 | 의미 |
|---|---|
| `source_profile = security` | Security Account credential을 사용 |
| `role_arn` | Target Account에서 Assume할 IAM Role |
| `region` | 기본 리전 |
| `output` | 출력 형식 |

---

### AssumeRole 확인

다음 명령어로 Target Account Role을 정상적으로 Assume했는지 확인한다.

```bash
>aws --profile target_security sts get-caller-identity

{
    "UserId": "AROAIKRY5GULQLYOGRMNS:botocore-session-1778400249",
    "Account": "653711331788",
    "Arn": "arn:aws:sts::653711331788:assumed-role/security/botocore-session-1778400249"
}
```

여기서 중요한 부분은 `"Account": "653711331788"`이다.



이제 현재 AWS CLI 명령은 Security Account가 아니라 **Target Account 권한 컨텍스트**에서 실행된다.

또한 ARN을 보면 다음과 같다.

```text
arn:aws:sts::653711331788:assumed-role/security/...
```

이는 IAM User로 직접 로그인한 것이 아니라, STS를 통해 Target Account의 `security` Role을 Assume한 상태라는 의미다.

---

### Account ID 구분

이번 실습에서 Account ID는 매우 중요하다.

| Account ID | 의미 |
|---|---|
| `322079859186` | Security Account |
| `653711331788` | Target Account |

따라서 `sts get-caller-identity` 결과에서 Account ID가 `322079859186`이면 Security Account에 있는 것이고, `653711331788`이면 Target Account context에 있는 것이다.

---

### Target Account S3 버킷 확인

Target Account Role을 Assume한 상태에서 다음 명령어를 실행한다.

```bash
aws --profile target_security s3 ls
```

그러면 Attacker 코스에서 봤던 Level별 S3 버킷들이 확인된다.

이는 현재 명령이 Target Account의 권한으로 실행되고 있다는 뜻이다.

---

### 보안 관점


보안팀은 각 AWS 계정에 직접 IAM User를 만들기보다는, 중앙 Security Account에서 각 계정의 보안 조사용 Role을 Assume하는 방식을 사용한다.

장점은 다음과 같다.

- 장기 Access Key 사용 감소
- 중앙 보안 운영 가능
- CloudTrail에 AssumeRole 기록이 남음
- 계정 간 접근 권한을 명확히 통제 가능
- Incident Response 수행이 쉬움

하지만 공격자도 AssumeRole 가능한 권한을 발견하면 이를 lateral movement에 사용할 수 있다.

따라서 Defender 입장에서는 CloudTrail에서 **AssumeRole** 이벤트를 주의 깊게 봐야 한다.




## Objective 3: 로그 분석 준비

### 요약

**CloudTrail JSON 로그에서 필요한 필드를 추출하여 공격 타임라인을 분석할 준비를 하는 단계**

CloudTrail 로그는 JSON 형식이지만, 원본 그대로 보면 양이 많고 구조가 복잡하다.  
따라서 `jq`를 사용해 필요한 필드만 추출한다.

---

### jq란?

`jq`는 JSON 데이터를 필터링하고 가공하는 CLI 도구다.

CloudTrail 로그처럼 중첩된 JSON 데이터를 분석할 때 매우 유용하다.

(flaws2.cloud 에서 알려주는 명령어는 리눅스/맥에서 지원되는 명령어다. 나는 WSL을 사용했다.)

---

### CloudTrail 로그 구조

CloudTrail 로그는 대략 다음 구조를 가진다.

```json
{
  "Records": [
    {
      "eventTime": "2018-11-28T23:09:28Z",
      "eventName": "ListBuckets",
      "eventSource": "s3.amazonaws.com",
      "sourceIPAddress": "x.x.x.x",
      "userIdentity": {
        "type": "IAMUser",
        "arn": "arn:aws:iam::..."
      }
    }
  ]
}
```

실제 이벤트들은 `Records` 배열 안에 들어 있다.

따라서 jq로 CloudTrail 로그를 분석할 때는 보통 `.Records[]` 패턴을 사용한다.

이는 `Records` 배열 안의 이벤트를 하나씩 꺼낸다.

---

### 이벤트 이름만 추출

```bash
find . -type f -exec cat {} \; | jq '.Records[]|.eventName'
```

이 명령은 CloudTrail 로그에서 API 이벤트 이름만 추출한다.


```text
"GetObject"
"GetObject"
"ListBuckets"
"AssumeRole"
"AssumeRole"
```

여기서 이벤트 이름은 실제로 호출된 AWS API를 의미한다.


| 이벤트 | 의미 |
|---|---|
| `GetObject` | S3 객체 조회 |
| `ListBuckets` | S3 버킷 목록 조회 |
| `AssumeRole` | IAM Role 임시 위임 |
| `GetDownloadUrlForLayer` | ECR 이미지 레이어 다운로드 URL 요청 |

---

### 시간과 이벤트 이름 함께 보기

이벤트 이름만 보면 순서를 알기 어렵다.  
따라서 `eventTime`을 함께 추출한다.

```bash
find . -type f -exec cat {} \; | jq -cr '.Records[]|[.eventTime, .eventName]|@tsv' | sort

2018-11-28T22:31:59Z    AssumeRole
2018-11-28T22:31:59Z    AssumeRole
2018-11-28T23:02:56Z    GetObject
2018-11-28T23:02:56Z    GetObject
2018-11-28T23:02:56Z    GetObject
...
```


이렇게 하면 이벤트를 시간순으로 정렬하여 볼 수 있다.

각 옵션의 의미는 다음과 같다.

| 옵션 | 의미 |
|---|---|
| `-c` | compact output, 한 줄 출력 |
| `-r` | raw output, 따옴표 제거 |
| `@tsv` | tab-separated values 형식 출력 |
| `sort` | 시간순 정렬 |


---

### 조사에 유용한 필드 추출

다음 명령은 CloudTrail 분석에 필요한 핵심 필드를 한 번에 뽑는다.

```bash
find . -type f -exec cat {} \; | jq -cr '.Records[]|[.eventTime, .sourceIPAddress, .userIdentity.arn, .userIdentity.accountId, .userIdentity.type, .eventName]|@tsv' | sort
```

추출되는 필드는 다음과 같다.

| 필드 | 의미 |
|---|---|
| `eventTime` | 이벤트 발생 시간 |
| `sourceIPAddress` | 요청 출발지 |
| `userIdentity.arn` | 요청 주체 ARN |
| `userIdentity.accountId` | 요청 주체 계정 ID |
| `userIdentity.type` | 요청 주체 유형 |
| `eventName` | 호출된 AWS API |

이 정보는 Defender 관점에서 가장 기본적인 질문에 답하기 위해 필요하다.

```text
누가?
언제?
어디서?
무엇을 했는가?
```

---

### userIdentity.type 해석

CloudTrail 분석에서 **userIdentity.type**은 매우 중요하다.

대표적인 값은 다음과 같다.

| type | 의미 |
|---|---|
| `IAMUser` | IAM User가 직접 API 호출 |
| `AssumedRole` | STS AssumeRole로 발급받은 임시 권한 사용 |
| `AWSService` | AWS 서비스가 내부적으로 호출 |
| `AWSAccount` | 다른 AWS 계정 주체 |
| `Anonymous` 또는 `ANONYMOUS_PRINCIPAL` | 인증되지 않은 요청 |

---

### AWSService 이벤트

로그에는 `AWSService` 타입 이벤트도 많이 등장한다.

예를 들어 ECS나 Lambda는 실행 과정에서 Role을 Assume한다.

```json
"userIdentity": {
  "type": "AWSService",
  "invokedBy": "ecs-tasks.amazonaws.com"
}
```

이런 이벤트는 사람이 직접 실행한 명령이라기보다 AWS 서비스가 정상 동작하면서 발생한 이벤트다.


```text
ECS Task가 Task Role을 Assume
Lambda가 Execution Role을 Assume
```

다만 이것이 무조건 무시해도 된다는 의미는 아니다.

공격자가 ECS나 Lambda를 악용한 경우, 그 이후 발급된 임시 credential이 외부에서 사용될 수 있다.

따라서 `AWSService` 이벤트 자체보다, 그 이후의 credential 사용 흐름을 확인하는 것이 중요하다.

---

### ANONYMOUS_PRINCIPAL

`ANONYMOUS_PRINCIPAL`은 인증되지 않은 요청을 의미한다.

이번 실습에서는 주로 public S3 객체를 웹 브라우저로 접근한 흔적으로 볼 수 있다.

구분할 때는 `userAgent`를 보면 된다.



| userAgent | 의미 |
|---|---|
| `Chrome`, `Mozilla` | 웹 브라우저 접근 |
| `aws-cli` | AWS CLI 사용 |
| `Botocore` | Python SDK 또는 AWS CLI 내부 라이브러리 |
| `Boto3` | Python SDK 사용 |

즉 `ANONYMOUS_PRINCIPAL`이면서 `userAgent`가 브라우저라면, S3 public object를 브라우저로 조회한 행위일 가능성이 높다.



## Defender 관점 핵심

Objective 1~3은 공격을 직접 추적하기 전의 준비 단계다.


- CloudTrail 로그가 어디에 저장되는지 이해
- Security Account와 Target Account의 역할 구분
- Cross-account AssumeRole 구조 이해
- CloudTrail JSON 구조 이해
- jq로 이벤트를 시간순 정리
- `IAMUser`, `AssumedRole`, `AWSService`, `Anonymous` 구분

이후 단계에서는 이 데이터를 기반으로 실제 공격자가 어떤 순서로 행동했는지 분석하게 된다.
